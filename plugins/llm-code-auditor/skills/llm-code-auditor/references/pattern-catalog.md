# LLM Code Pattern Catalog

Use this reference when auditing code that may be generated by an LLM or coding agent. It expands the high-confidence checklist in `SKILL.md`.

## Research Basis

- Tambon et al., "Bugs in Large Language Models Generated Code: An Empirical Study" sampled 333 LLM-generated bugs and identified recurring bug patterns: misinterpretations, syntax errors, silly mistakes, prompt-biased code, missing corner cases, wrong input type, hallucinated object, wrong attribute, incomplete generation, and non-prompted consideration. https://arxiv.org/abs/2403.08937
- Velasco et al., "How Propense Are Large Language Models at Producing Code Smells?" argues that accuracy benchmarks miss generated-code quality and introduces CodeSmellEval to measure code smell propensity. Their case study found models producing smells such as simplifiable conditions and mergeable `isinstance` checks. https://arxiv.org/abs/2412.18989
- Cordeiro et al., "An Empirical Study on the Code Refactoring Capability of Large Language Models" found LLM refactoring can reduce systematic smells, but human developers do better on complex, context-dependent issues. Use LLM-style refactoring for mechanical cleanup, then apply human context. https://arxiv.org/abs/2411.02320
- A large-scale study of transformer-based code generation found generated outputs can contain problematic code smells and emphasized not accepting generated code as-is. https://s2e-lab.github.io/preprints/scam22-preprint.pdf

## Smell Families And Fixes

### 1. Single-use abstraction

Detect:
- Interface/base class with exactly one implementation.
- Helper called once.
- Wrapper class forwarding all methods.
- Dependency-injection seam used only for an in-memory collaborator.

Fix:
- Inline helper into caller.
- Replace interface with concrete type unless it protects an external service, test seam, plugin contract, or stable API.
- Collapse wrapper into the module that owns the behavior.

Counterexample:
- Keep a single implementation when it represents a boundary likely to be mocked or replaced now: payment provider, clock, filesystem, database, network client.

### 2. Naming inflation

Detect names ending in `Manager`, `Service`, `Processor`, `Handler`, `Provider`, `Factory`, `Controller`, `Engine`, `Coordinator`, `Orchestrator`, `Resolver`, `Executor`, `Helper`.

Fix:
- Rename by owned domain concept: `InvoiceTotals`, `PasswordResetEmail`, `MailDraft`, `CheckoutSession`.
- Split mixed responsibilities before renaming.

### 3. Utility dumping

Detect files or modules named `utils`, `helpers`, `common`, `shared`, `base`, `misc`, `general`.

Fix:
- Move code beside the caller or owning domain module.
- Split truly shared code by concept: `dates`, `money`, `mail_addresses`, `permissions`.

### 4. Narration comments

Detect comments that say what the next line already says: "increment counter", "return result", "check if user exists".

Fix:
- Delete narration.
- Keep comments only for why: protocol quirks, ordering constraints, security assumptions, performance tradeoffs, bug references.

### 5. Pass-through layers

Detect:
- `foo(a, b) { return bar(a, b); }`
- classes that only delegate all methods unchanged.
- API endpoints that call services that call repositories with identical argument lists and no invariant added.

Fix:
- Remove the layer.
- If the layer is required by framework routing, move validation or mapping there so it earns its existence.

### 6. Speculative extensibility

Detect:
- plugin registries with one plugin.
- strategy maps with one strategy.
- event buses with one subscriber.
- generic options objects with unused fields.
- abstract factories without product families.

Fix:
- Use direct calls and concrete data.
- Keep a small explicit branch only when the second case exists now.

### 7. AI symmetry

Detect:
- Files with suspiciously similar line counts and structure.
- Repeated CRUD skeletons with no domain-specific constraints.
- Mirrored method names across unrelated concepts.

Fix:
- Consolidate the true common core.
- Specialize where domain invariants differ.
- Prefer a small table-driven map for boring repeated mappings; avoid a giant framework.

### 8. Excessive defensive programming

Detect:
- Null checks impossible by type system or constructor invariant.
- Duplicate validation at controller, service, repository, and model layers.
- `try/catch` blocks that log and rethrow.
- default values that hide programmer error.

Fix:
- Validate once at trust boundary.
- Encode invariant in type, constructor, parser, or schema.
- Let exceptions propagate unless local context can recover or add meaningful detail.

### 9. Over-fragmentation

Detect:
- one tiny class per file with no reuse.
- deep directories containing one file each.
- enterprise nouns copied from architecture diagrams.

Fix:
- Merge cohesive code into a single module.
- Keep folders aligned with domain boundaries, not design-pattern names.

### 10. Generic abstraction language

Detect:
- `entity`, `item`, `object`, `data`, `info`, `payload`, `context` used where the domain has precise nouns.
- `processData`, `handleRequest`, `executeTask`, `performAction`, `doStuff`.

Fix:
- Rename from schemas, events, commands, UI labels, logs, and user-visible workflows.

### 11. Hallucinated APIs and attributes

Detect:
- method/property names that resemble a library but do not exist.
- code that compiles only if a guessed object shape is true.
- broad `any`, `dict`, `unknown`, or reflection used to silence the mismatch.

Fix:
- Check official docs, local type definitions, generated clients, or runtime introspection.
- Replace guessed shapes with typed adapters at boundaries.

### 12. Prompt-biased code

Detect:
- sample values hard-coded from the prompt.
- code path works only for the demo input.
- tests repeat the implementation examples instead of independent behavior.

Fix:
- Generalize from product requirements, not prompt wording.
- Add edge tests that vary names, counts, ordering, duplicates, and invalid input.

### 13. Faux robustness

Detect:
- retries without idempotency.
- catch-all exception handling.
- fallback branches that silently produce wrong results.
- logging sensitive values.

Fix:
- Define failure semantics.
- Make retries bounded, idempotent, and observable.
- Fail closed for auth/security.

### 14. Context-insensitive optimization

Detect:
- memoization/cache added without invalidation.
- parallelism added without concurrency limits.
- streaming/batching added without backpressure.
- micro-optimizations in non-hot paths while algorithmic cost remains bad.

Fix:
- Measure or reason from input size.
- Prefer simpler algorithmic improvements and existing framework primitives.

## High-Quality Human Patterns

- Direct domain model: code structure follows product nouns and invariants.
- Fewer moving parts: obvious call graph, low indirection, small public API.
- Boundary discipline: parsing/validation/auth happen at trust boundaries; internal code receives validated shapes.
- Locality: a reader can see data, invariant, and behavior together.
- Strategic duplication: duplicate two lines when abstraction would hide meaning; abstract only after real repeated pressure.
- Tests at risk points: edge cases, error semantics, concurrency/idempotency, integration contracts.
- Deletion mindset: the best refactor often removes unused options, generic layers, and impossible states.
- Non-obvious connection: reuse existing domain invariants, schemas, state machines, or framework lifecycle hooks instead of inventing parallel machinery.

## Review Questions

- What behavior would break if this abstraction disappeared?
- Is this name a domain concept or a vague job title?
- Is validation happening at the first untrusted boundary or everywhere?
- Are tests proving behavior or just snapshotting generated structure?
- Does this code solve the real product case or just the prompt example?
- What invariant can make this branch impossible?
- What code can be deleted after moving behavior closer to the owner?
